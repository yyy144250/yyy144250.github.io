session,cookie,token是什么

圆圆圆 · 2020-3-5 · 次阅读

前言

之前虽然用到了cookie,session和token，以为理解了用法，但是当有人问起时，却又说不出来，所以决定全方位的理解一下这几个名词的含义和用法，作为自己的总结和学习，也作为分享,如有错误，希望能指出。

通俗的理解

看到过一个和有趣的例子
学校里的奶茶店有喝5杯奶茶免费赠一杯奶茶的优惠，然而一次性消费5杯奶茶的机会微乎其微，这时就需要某种方式来纪录某位顾客的消费数量

如果协议本身支持状态

该店的店员很厉害，能记住每位顾客的消费数量，只要顾客一走进奶茶店，店员就知道该怎么对待了。这种做法就是协议本身支持状态
可惜我们的协议本身是无状态的

Cookie的方式

发给顾客一张卡片，上面记录着消费的数量，一般还有个有效期限。每次消费时，如果顾客出示这张卡片，则此次消费就会与以前或以后的消费相联系起来。这种做法就是在客户端保持状态。

Session的方式

发给顾客一张会员卡，除了卡号之外什么信息也不纪录，每次消费时，如果顾客出示该卡片，则店员在店里的纪录本上找到这个卡号对应的纪录添加一些消费信息。这种做法就是在服务器端保持状态。

Token的方式

发给顾客一张会员卡，除了卡号之外什么信息也不纪录，每次消费时，顾客需要出示卡片，同时还需要输入密码，如果密码正确则可以找到记录，添加消费信息

Cookie

什么是Cookie?

HTTP协议本身是无状态的。什么是无状态呢，即服务器无法判断用户身份。Cookie实际上是一小段的文本信息（key-value格式）。客户端向服务器发起请求，如果服务器需要记录该用户状态，就使用response向客户端浏览器颁发一个Cookie。客户端浏览器会把Cookie保存起来。当浏览器再请求该网站时，浏览器把请求的网址连同该Cookie一同提交给服务器。服务器检查该Cookie，以此来辨认用户状态。

Cookie的存储

以文件的方式存储在硬盘空间上的长期性的cookie:比如登录时填写的一些信息会被以文件的方式放在客户端的硬盘上。

Cookie发送过程

  • 客户端发送request给服务端
  • 服务端发现是第一次登陆的用户，返回response携带set-cookie给客户端
  • 客户端保存cookie，在下次请求时携带cookie
  • 服务端验证cookie，确认身份，返回response

Cookie的弊端

cookie 中的所有数据在客户端就可以被修改，数据非常容易被伪造，那么一些重要的数据就不能存放在 cookie 中了，而且如果 cookie中数据字段太多会影响传输效率。为了解决这些问题，就产生了 session，session 中的数据是保留在服务器端的

Session

什么是Session

session是一种服务器端的信息管理机制，它把这些文件信息以文件的形势存放在服务器的硬盘空间上。本质上session和cookie没有什么区别，只不过session存放在服务端，cookie存放在服务端
不同的Web框架下Session的实现机制都是不一样的。但是原理都是大同小异的，这里普遍应用的机制是通过Cookie来存储一个会话的票据（也就是 SessionID）

Session的发送过程

Session存储在Cookie中，所以发送过程大致和cookie相同

  • 浏览器第一次访问服务器会在服务器端生成一个session，有一个sessionid和它对应，类似于键值对key：value
  • 在Session被创建之后，就可以调用Session相关的方法往Session中增加内容了，而这些内容只会保存在服务器中，比如redis 或 memcached 等缓存中
  • 发送给客户端Session id；当客户端再次发送请求的时候，会将这个Session id带上，服务器接受到请求之后就会依据Session id找到相应的Session，从而再次使用之。

Session的安全性

Session生成规则是根据hash_func散列来生成的，相关的参数有：

  • 客户端IP
  • 当前时间（秒）
  • 当前时间（微妙）
  • PHP自带的随机数生产器
    从生成规则来看，如果想要破解sessionid是几乎不可能的。所以session是很安全的

Session的发送过程

  • 客户端第一次发送request给服务端，服务端为其开辟session存储空间
  • 服务端返回response携带set-cookie，sessionid存在返回的cookie中
  • 客户端保存cookie，下次请求时携带存有session_id的cookie发送request
  • 服务端验证cookie中的session_id并对session进行一系列操作。并返回response

cookie，session_id的发送过程注意事项

首先cookie信息是由浏览器自动为我们发送的，但是由于同源策略的限制，当我们的前后端不在同一域名下时，会导致cookie无法发送，所以需要我们配置允许跨域。常见的方法有使用cors，添加withCredentials = true。允许浏览器发送非同源请求时携带cookie，如vue中

var instance =axios.create({
    withCredentials:true
    })

Token

什么是token

token是计算机术语：令牌，令牌是一种能够控制站点占有媒体的特殊帧，以区别数据帧及其他控制帧。token其实说的更通俗点可以叫暗号，在一些数据传输之前，要先进行暗号的核对，不同的暗号被授权不同的数据操作。
说白了本质上也是用于身份验证的东西，那么为什么有了session，还又出现了token？自然是因为session不好，token好

token的好处

  • Token可以是无状态的，可以在多个服务器见共享
  • Token支持手机端访问
  • Token完全是应用管理，所以它可以避开同源策略，cookie是不允许跨域访问，token则不存在
  • Token可以避免CSRF攻击

token的无状态

当使用session时，大家都只需要把自己的sessionid存在客户端，可是服务端却要存每一个人的sessionid，如果人多了，就要存不知道多少个，同时，如果我现在要用两个机器组成一个集群，我在A上面登陆了，sessionid保存在上面，下一次我的请求被送到了B上，怎么验证？当让，万能的大佬们支招说可以，可以把sessionid集中储存在同一个地方，让所有机器都访问这个地方的数据，这样就解决了，但是负责sessionid的这个机器挂了的话，不就完蛋了？

所以既然存session有很多的麻烦，我们就不保存sessionid了，那我们怎么验证身份并保证身份正确呢。有一个方法，我们对他这个用户名进行一个签名，学名叫信息摘要算法。，比如用一个SHA算法，用一个只有我自己知道的密钥，进行加密。然后我们不保存这个token，我们等客户端把数据发送过来后，把他的用户名进行同样的加密，如果和他传过来的token值相同，则说明这个用户名没被更改过。

  • 比如有一个字段user=yyy，我们将这个值和我们服务端的一个秘密字符串secret_string进行hash，比如sha1(user=yyy+secret_string)= ‘4850a42e3bc0d39c978770392cbd8dc2923e3d1d’
  • 发送的时候同时发送user=yyy以及user_sign=’4850a42e3bc0d39c978770392cbd8dc2923e3d1d’，当客户端接收到时，用同样的算法对着一个字符串进行相同的加密，如果结果和user_sign相同，则说明数据没有被篡改，而加密所需的秘密字符串只有我们客户端才知道，别人是无法伪造出来user_sign的。

从token的整个过程我们可以看出我们发送的token信息不需要在服务端存储什么验证信息，只需要存储一个秘密字符串和加密算法，这大大减轻了服务端的压力，这样服务端就有余力做更多其他的事情。

token可以防止CSRF攻击

CSRF

什么是CSRF

CSRF攻击的全称是跨站请求伪造（ cross site request forgery)，是一种对网站的恶意利用。简单点讲就是，恶意网站(攻击者)盗用了你的身份，以你的名义向信任网站发送恶意请求。 CRSF能做的事情包括利用你的身份发邮件、发短信、进行交易转账等，甚至盗取你的账号

CSRF过程

CSRF攻击过程大体上为

  • 受害者登陆a.com,并保留了登陆凭证cookie
  • 攻击者引诱受害者访问者访问b.com，如邮件
  • b.com向a.com的js中会发送了一个请求a.com/act=xx让受害者去访问a.com,浏览器会默认带上a.com的cookie
  • a.com收到请求后验证了cookie，确实是受害者的cookie误认为是受害者发送的cookie
  • a.com以受害者的名义执行了act=xx
  • 攻击完成，攻击者虽然没有获取cookie的内容，但是使用了受害者的cookie

以银行网站为例（原例子博客https://www.cnblogs.com/hyddd/archive/2009/04/09/1432744.html）

GET

银行网站A，它以GET请求来完成银行转账的操作，如：http://www.mybank.com/Transfer.php?toBankId=11&money=1000

危险网站B，它里面有一段HTML的代码如下：

<img src=http://www.mybank.com/Transfer.php?toBankId=11&money=1000>

这样当用户访问B网站时，便会自动携带A网站的cookie执行A网站的转钱操作，并且整个过程身份验证的都是他自己

POST

银行网站这一次不用GET了，改使用POST了

<form action="Transfer.php" method="POST">
<p>ToBankId: <input type="text" name="toBankId" /></p>
<p>Money: <input type="text" name="money" /></p>
<p><input type="submit" value="Transfer" /></p>
</form>

所以为什么Token可以防止cors攻击呢

TOken与CORS

一般情况下我们收到服务端的token之后是会把token存在local storage或者cookie中，前面的session_id存储再cookie中导致cookie一旦被盗用，那么cookie中的session_id也会同样的被盗用进行了身份验证，那么token存储再cookie中时为什么不会被盗用呢？
这是因为，在信任网站的js中，会向服务器传递参数token，不是通过Cookie传递的，若恶意网站要伪造用户的请求，也必须伪造这个token，否则用户身份验证不通过。但是，同源策略限制了恶意网站不能拿到信任网站的Cookie内容，只能使用，所以就算是token是存放在Cookie中的，恶意网站也无法提取出Cookie中的token数据进行伪造。也就无法传递正确的token给服务器，进而无法成功伪装成用户了。
即我们的token是存放在cookie中，但是我们的cookie进行检验时，并不是使用的cookie中的token，cookie中的token只是用来存储的，它能够使用我们的cookie，却无法知道我们cookie里面的值，无法取出里面的token进行验证，cookie又不会直接用里面的token验证，自然就不会被CSRF了

token的使用过程

  • 服务端签发的下来的token，将其存储到cookie或者localstorage中
    localStorage.setItem('token', data.data.token);

  • 每次发送请求都携带本地的token去发送

    config => {        
    if (localStorage.token) {  // 判断是否存在token，如果存在的话，则每个http header都加上token             
        config.headers['authorization'] = 'authorization' + localStorage.token;        
    }        
    return config    
},

  • Token过期

    If(data.code==2){
   window.location.href=“login.html”
}
  • Token的发送方式有很多，此为放到请求的header中,除此之外还有遍历dom树为每个a标签添加token等方式

refresh_token和acess_token

用于验证的token有效时间应该尽可能短，但是有效时间过短则需要用户去进行不断地登陆，而token允许我们一些附加属性，当采用有效的方式请求访问令牌时，授权服务器则会返回一个授权令牌（access token）和一个可选的刷新令牌(refresh token),同时返回给客户端还有其他一些附加属性，比如用户姓名，编号等

  • 访问令牌 access_token (必需)
  • 令牌类型 token_type (必须)
  • 超时时间 expires_in (推荐)
  • 刷新令牌 refresh_token （可选）
  • 作用域 scope（可选）

相应的过程大致为

  • 客户端向服务端发送请求，服务端返回refresh token和accesstoken
  • 客户端再次发送请求时携带access token进行验证身份
  • 如果服务端发现acesstoken已经过期，则返回消息说过期了，而refreshtoken没有过期，则自动刷新acesstoken
  • 如果refreshtoken也过期了，则返回登陆页面，让他重新登陆

这样当acess泄露时，由于有效期限很短，所以可以在一定程度上较小造成的不良后果。
